# ASSIST_RULES

## 0. 목적

FOM Assist는 정규화된 원천데이터를 입력으로 받아 FOM File set의 QPR 파일로 변환하는 GUI 기반 전처리기이다.

본 문서는 FOM Assist가 어떤 입력을 받으며, 어떤 규칙으로 사용자와 상호작용하고, 어떤 기준으로 QPR 변환을 수행해야 하는지를 정의한다.

---

## 1. 범위

FOM Assist의 범위는 다음으로 한정한다.

- 입력 대상은 정규화된 테이블형 원천데이터이다.
- 입력 파일 형식은 CSV 또는 Excel을 기본 대상으로 한다.
- 원천데이터는 이미 행/열 구조가 안정된 상태라고 가정한다.
- 컬럼명은 프로젝트별로 다를 수 있으나, 데이터는 표 형식이어야 한다.
- FOM Assist는 원천데이터를 QPR 구조에 맞게 매핑하고 변환하는 역할을 수행한다.

다음 항목은 범위에 포함하지 않는다.

- 수기 문서 직접 해석
- 병합 셀, 다중 헤더, 자유형식 보고서의 구조 복원
- 정규화되지 않은 엑셀 문서 정리
- OCR 또는 비정형 문서 판독

즉, 정규화 이전 단계는 별도 애플리케이션 또는 별도 프로세스가 담당한다.

---

## 2. 기본 전제

- 프로젝트마다 원천데이터 컬럼명은 다를 수 있다.
- 사용자는 GUI에서 원천 컬럼과 QPR 표준 컬럼을 매핑한다.
- 매핑 결과는 프로젝트별 Rule로 저장하고 재사용한다.
- QPR 구조 및 해석 규칙은 `QPR_RULES.md`를 따른다.

---

## 3. 입력 데이터 규칙

입력 데이터는 다음 조건을 만족해야 한다.

- 첫 행 또는 지정된 헤더 행에 컬럼명이 존재해야 한다.
- 각 행은 하나의 관측 단위 또는 작업 관련 기록이어야 한다.
- 컬럼 단위 의미가 일관적이어야 한다.
- 데이터는 CSV, XLS, XLSX 등의 표 형식이어야 한다.

빈 값 처리 원칙은 다음과 같다.

- 빈 문자열은 NULL로 본다.
- 공백 문자열은 trim 후 NULL 여부를 판단한다.
- 숫자형 데이터도 최초에는 문자열로 읽고 후처리에서 변환한다.

## 4. QPR 변환 대상 컬럼

FOM Assist는 `QPR_RULES.md`에 정의된 다음 19개 QPR 표준 컬럼을 대상으로 매핑을 수행한다.

### 4.1 작업 식별 및 공정 계층 컬럼

1. `일자`: 작업 수행 날짜
2. `Shift`: 주/야
3. `대분류`: 생산 라인 또는 공정 계층 Level1
4. `중분류`: 공정 단계 또는 공정 계층 Level2
5. `소분류`: 제품 타입 또는 공정 계층 Level3
6. `설비`: 사용 설비
7. `제품`: 생산 제품
8. `작업자`: 작업 수행자

### 4.2 대표 생산 정보 컬럼

9. `실적수량`: 실제 생산량
10. `계획수량`: 계획 생산량
11. `작업시간`: 총 작업시간
12. `효율`: 효율 값
13. `C/T`: 사이클 시간

위 대표 생산 정보는 QPR 해석 시 Main Row의 값을 기준으로 사용한다.

### 4.3 요인 정보 컬럼

14. `비가동`: 비가동 요인
15. `비가동시간`: 비가동 시간
16. `부적합`: 부적합 발생 요인
17. `부적합수량`: 부적합 발생 수량
18. `불량`: 불량 발생 요인
19. `불량수량`: 불량 발생 수량

요인 정보 컬럼은 Main Row와 Detail Row를 포함한 동일 Key 그룹 전체에서 누적 수집 대상이 된다.

- 불량
- 불량수량

## 6. 공정 계층 처리 규칙

대분류, 중분류, 소분류는 프로젝트마다 실제 의미가 다를 수 있다.

예:

- 라인 / 공정 / 세부공정
- 공장 / 라인 / 설비군
- 대분류 / 중분류 / 소분류

따라서 FOM Assist는 이 3개를 고정 도메인 의미로 강제하지 않고, QPR의 공정 계층 슬롯으로 처리한다.

- 대분류 = process level 1
- 중분류 = process level 2
- 소분류 = process level 3

GUI에서는 QPR 표준 컬럼명을 유지하되, 프로젝트별 의미 라벨을 부가적으로 저장할 수 있다.

## 7. 매핑 방식

각 QPR 컬럼은 다음 세 가지 방식 중 하나로 지정된다.

### 7.1 column

원천데이터 컬럼과 직접 연결한다.

### 7.2 default

기본값을 사용한다.

### 7.3 blank

공백으로 둔다.

### 7.4 다중 슬롯 요인 매핑 (cause_mapping)

요인 컬럼(비가동/부적합/불량)은 위 세 가지 방식과 별도로 다중 슬롯 매핑을 사용한다.

원천 데이터에 요인이 여러 컬럼에 분산된 경우(`비가동요인1/시간1`, `비가동요인2/시간2` 등), 슬롯을 추가하여 매핑한다.

- 슬롯은 (요인 컬럼명, 수량·시간 컬럼명) 쌍으로 구성된다.
- **슬롯 0**: 해당 요인 유형의 첫 번째 쌍 → Main Row에 직접 기록한다.
- **슬롯 1 이상**: 동일 복합 Key를 가진 Detail Row를 추가로 생성한다.
  - Detail Row는 Key 컬럼(일자·Shift·대·중·소분류·설비·제품·작업자)만 복사하고, 해당 요인·수량 컬럼만 채운다.
  - 생산 수량 컬럼(실적수량·계획수량·작업시간 등)은 빈값으로 둔다.
- 요인 값과 수량 값이 모두 비어 있는 슬롯은 출력에서 생략한다.
- 3개 요인 유형(비가동/부적합/불량)은 각각 독립적으로 슬롯을 가진다.

**예시 — 비가동 2슬롯 + 불량 1슬롯:**

| 구분 | 실적수량 | 비가동 | 비가동시간 | 불량 | 불량수량 |
|------|----------|--------|------------|------|----------|
| Main Row | 100 | 품질 | 2 | 찍힘 | 3 |
| Detail Row | (빈값) | 설정변경 | 1 | (빈값) | (빈값) |

**저장 형식 (`mapping_rules.json` v2):**

```json
{
  "version": 2,
  "rules": { "...비요인 13개 컬럼..." },
  "cause_mapping": {
    "downtime":      [{"cause_src": "비가동요인1", "qty_src": "비가동시간1"},
                      {"cause_src": "비가동요인2", "qty_src": "비가동시간2"}],
    "nonconformity": [{"cause_src": "부적합요인1", "qty_src": "부적합수량1"}],
    "defect":        []
  }
}
```

v1 포맷(`mapping_rules.json` version 없음 또는 version=1)은 로드 시 자동으로 단일 슬롯 cause_mapping으로 마이그레이션된다.

## 8. 프로젝트별 Rule 저장

매핑 Rule은 프로젝트별로 저장한다.

저장 대상은 다음을 포함한다.

- 프로젝트명
- 입력 파일 정보
- 헤더 행 정보
- QPR 비요인 13개 컬럼 매핑 결과 (column/default/blank)
- 요인 6개 컬럼의 다중 슬롯 매핑 (`cause_mapping`)
- 기본값 규칙
- 공백 허용 규칙

저장 포맷은 JSON v2를 기본으로 한다. v2에서 요인 6개 컬럼은 `rules`에 blank로 저장하고, `cause_mapping` 키에 별도 관리한다.

FOM Assist가 QPR 파일을 생성할 때는, 해당 QPR를 만들기 위해 사용한 프로젝트별 설정을 `config.json`으로 함께 생성하거나 갱신해야 한다.

즉, QPR 산출물은 다음 두 요소를 한 세트로 본다.

- QPR 결과 파일
- 프로젝트별 `config.json`

`config.json`은 단순 부가 파일이 아니라, 다음 실행 시 재사용 가능한 프로젝트 설정이자 변환 이력을 재현하기 위한 기준 파일이다.

### 8.1 `config.json`의 역할

`config.json`은 프로젝트별 원천데이터를 현재 QPR 파서가 해석할 수 있도록 연결해 주는 설정 파일이다.

현재 기준 파일은 [`projects/Project_A/config.json`](d:/OpenFOMs/projects/Project_A/config.json)이며, 다음과 같은 항목 구조를 가진다.

- `delimiter`
- `date_col`
- `shift_col`
- `process_hierarchy`
- `machine_col`
- `product_col`
- `worker_col`
- `plan_qty_col`
- `actual_qty_col`
- `work_time_col`
- `efficiency_col`
- `cycle_time_col`
- `downtime_col`
- `downtime_time_col`
- `nonconformity_col`
- `nonconformity_qty_col`
- `defect_col`
- `defect_qty_col`

각 항목의 의미는 다음과 같다.

- `delimiter`: 원천 CSV 파일의 구분자
- `date_col`: 작업일자에 해당하는 원천 컬럼명
- `shift_col`: Shift에 해당하는 원천 컬럼명
- `process_hierarchy`: 공정 계층(Level1, Level2, Level3)에 해당하는 원천 컬럼명 목록
- `machine_col`: 설비에 해당하는 원천 컬럼명
- `product_col`: 제품에 해당하는 원천 컬럼명
- `worker_col`: 작업자에 해당하는 원천 컬럼명
- `plan_qty_col`: 계획수량에 해당하는 원천 컬럼명
- `actual_qty_col`: 실적수량에 해당하는 원천 컬럼명
- `work_time_col`: 작업시간에 해당하는 원천 컬럼명
- `efficiency_col`: 효율에 해당하는 원천 컬럼명
- `cycle_time_col`: C/T에 해당하는 원천 컬럼명
- `downtime_col`: 비가동 요인에 해당하는 원천 컬럼명
- `downtime_time_col`: 비가동시간에 해당하는 원천 컬럼명
- `nonconformity_col`: 부적합 요인에 해당하는 원천 컬럼명
- `nonconformity_qty_col`: 부적합수량에 해당하는 원천 컬럼명
- `defect_col`: 불량 요인에 해당하는 원천 컬럼명
- `defect_qty_col`: 불량수량에 해당하는 원천 컬럼명

FOM Assist는 QPR 생성 시 위 구조를 기본으로 하는 `config.json`을 프로젝트 폴더에 생성하거나 갱신해야 한다.

## 9. GUI 원칙

FOM Assist GUI는 짧은 시간 안에 직관적으로 매핑을 완료할 수 있어야 한다.

원칙은 다음과 같다.

- 한 화면 중심으로 동작해야 한다.
- 원천 컬럼과 QPR 컬럼의 대응 관계가 한눈에 보여야 한다.
- 단순 드롭다운 나열보다 시각적 연결이 우선된다.
- 필수 / 기본값 / 공백 허용 상태를 즉시 식별할 수 있어야 한다.
- 매핑 결과는 프로젝트별로 저장 및 재사용 가능해야 한다.

---

## 10. 매핑 화면 규칙

매핑 화면은 다음 요소를 포함한다.

- 좌측 스크롤 영역: 원천데이터 컬럼 목록
- 중앙 연결선 영역: 컬럼 간 연결선 (실시간 렌더링)
- 우측 스크롤 영역: QPR 컬럼 목록 (비요인 13개 + 요인 확장 패널 3개)
- 하단 또는 우측: 검증 결과와 미리보기

시각적 상태 규칙은 다음과 같다.

- 직접 매핑(column): 파란 실선
- 기본값(default): 회색 점선
- 공백: 선 없음
- 요인 슬롯 (요인 컬럼): 주황 파선
- 요인 슬롯 (수량/시간 컬럼): 주황 점선 (더 가늘게)
- 필수 누락: 빨간 강조
- 정상 완료: 초록 표시

요인 컬럼(비가동/부적합/불량)은 일반 QPRColumnWidget 대신 **확장 패널(CauseGroupPanel)**로 표시된다.

- 패널 헤더에 `[+ 슬롯]` 버튼이 있으며, 슬롯을 동적으로 추가·삭제할 수 있다.
- 각 슬롯은 (요인 컬럼 선택 콤보, 수량/시간 컬럼 선택 콤보, [×] 삭제 버튼)으로 구성된다.
- 요인 수량 컬럼(`비가동시간`, `부적합수량`, `불량수량`)은 우측 목록에서 별도 항목으로 표시하지 않고 패널 내부 슬롯으로 처리한다.

## 11. QPR 구조 인식 규칙

FOM Assist는 QPR를 단순한 1행 1레코드 구조로 다루지 않는다.

QPR는 동일 Key를 가진 행들의 그룹 구조로 해석한다.

- 하나의 작업 단위는 동일 Key 그룹이다.
- 하나의 그룹은 Main Row 1개와 Detail Row 0개 이상으로 구성될 수 있다.
- 대표 생산 정보는 Main Row에서 가져온다.
- 요인 정보는 Main Row와 Detail Row 전체에서 누적 수집한다.

세부 해석 규칙은 `QPR_RULES.md`를 따른다.

---

## 12. 요인 정보 표시 규칙

비가동, 부적합, 불량은 각각 0개 이상 존재할 수 있다.

따라서 GUI는 요인 컬럼을 단일값처럼 오해하게 만들면 안 된다.

미리보기에서는 다음을 보여줄 수 있어야 한다.

- 작업 그룹별 요인 건수
- 비가동 n건
- 부적합 n건
- 불량 n건

필요 시 사용자는 각 그룹을 펼쳐서 요인 상세 리스트를 볼 수 있어야 한다.

---

## 13. 검증 규칙

FOM Assist는 최소한 다음을 검증해야 한다.

- 필수 컬럼 매핑 여부
- 기본값 적용 컬럼 목록
- 공백 처리 컬럼 목록
- 변환 가능 여부

QPR 그룹 해석 시에는 추가로 다음을 확인할 수 있다.

- Main Row 없음
- Main Row 중복
- 숫자 변환 실패
- 날짜 해석 실패

검증 결과는 오류와 경고로 구분한다.

- 오류: 변환 불가
- 경고: 변환 가능하나 품질 이슈 존재

## 14. 출력

FOM Assist의 결과는 다음을 생성할 수 있어야 한다.

- QPR CSV 또는 내부 표준 구조
- 검증 결과 및 경고 로그
- 프로젝트별 매핑 Rule 파일
- 프로젝트별 `config.json`

QPR 출력이 성공하면, 동일 프로젝트 폴더 내에 QPR 결과와 함께 해당 변환 기준을 담은 `config.json`이 저장되어야 한다.

---

## 15. 설계 원칙

FOM Assist는 다음 원칙을 따른다.

- 복잡성보다 직관성을 우선한다.
- 비정형 문서 처리는 범위에서 제외한다.
- 프로젝트별 차이는 코드가 아니라 매핑 Rule로 흡수한다.
- QPR 구조 해석은 `QPR_RULES.md`와 일치해야 한다.
- 사용자는 짧은 시간 안에 QPR 생성까지 도달할 수 있어야 한다.
