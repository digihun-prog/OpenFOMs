
## 0. 파일 형식

QPR 파일은 CSV 형식으로 제공된다.

- 파일 확장자: .csv
- 인코딩: UTF-8 또는 EUC-KR (프로젝트별 상이할 수 있음)
- 구분자: 콤마(,) 기본 (프로젝트에 따라 탭 또는 세미콜론 가능)

### 처리 규칙

- 모든 컬럼 값은 문자열로 읽은 후, 필요 시 숫자형으로 변환한다.
- 숫자 컬럼(계획수량, 실적수량, 작업시간, 비가동시간, 부적합수량, 불량수량)은 후처리에서 형 변환한다.
- 빈 값("")은 NULL로 해석한다.
- 공백 문자열(" ")은 trim 후 NULL 여부를 판단한다.

## 2. Key 정의

QPR 데이터에서 하나의 작업 단위를 식별하는 Key는 다음 논리적 항목의 조합으로 정의한다.

- 일자 (date)
- Shift (shift)
- 공정 계층 (process hierarchy)
- 설비 (machine)
- 제품 (product)
- 작업자 (worker)

### 공정 계층 (Process Hierarchy)

공정 계층은 프로젝트별로 다음과 같이 가변적으로 정의된다.

- Level1 (예: 대분류, 라인)
- Level2 (예: 중분류, 공정)
- Level3 (예: 소분류, 세부공정 또는 제품타입)

각 Level의 컬럼명은 프로젝트 설정(config)에 따라 매핑된다.

예:
- 프로젝트 A: 대분류 / 중분류 / 소분류
- 프로젝트 B: 라인 / 공정 / 공정세부
- 프로젝트 C: Line / Process / Step

위 공정 계층 컬럼들은 동일 Key 판단 시 모두 포함된다.


## 3. 행 구조 정의 (Main Row vs Detail Row)

QPR 데이터에서 동일 Key를 가지는 여러 행은 하나의 작업 단위로 묶인다.

이때 각 행은 다음 두 가지 유형으로 구분된다.

### 3.1 Main Row (메인 행)

다음 조건을 만족하는 행을 Main Row로 정의한다.

- 계획수량 또는 실적수량에 값이 존재한다.

Main Row의 역할:
- 해당 작업 단위의 대표 데이터
- 생산량, 작업시간, 효율, C/T 등의 기준값 제공
- 비가동, 부적합, 불량 원인 정보가 존재할 경우 각 항목의 첫 번째 원인 정보를 함께 포함할 수 있다

주의사항:
- 동일 Key 내에서 Main Row는 원칙적으로 1개만 존재해야 한다.
- 2개 이상 존재할 경우 데이터 오류 가능성이 있으므로 경고 대상으로 처리한다.

### 3.2 Detail Row (추가 행)

다음 조건을 만족하는 행을 Detail Row로 정의한다.

- 계획수량과 실적수량이 모두 비어 있다.

Detail Row의 역할:
- Main Row에 포함되지 못한 추가 원인 정보를 표현하기 위한 행
- 비가동, 부적합, 불량 각각의 원인 수가 2개 이상일 경우 원인별로 추가된다

### 3.3 원인 정보 해석 규칙

비가동, 부적합, 불량 정보는 Main Row 또는 Detail Row 어느 곳에도 존재할 수 있다.

각 원인 항목은 "원인별 1행" 원칙으로 해석한다.

예:
- 비가동 원인 1개: Main Row 또는 Detail Row 1개에 기록 가능
- 비가동 원인 2개 이상: 첫 번째 원인은 Main Row에 포함될 수 있으며, 나머지는 원인별 Detail Row로 추가된다
- 부적합 원인 2개 이상: 원인별 Detail Row로 추가될 수 있다
- 불량 원인 2개 이상: 원인별 Detail Row로 추가될 수 있다

즉, 다음 항목들은 모두 원인 단위로 누적 수집한다.

#### (1) 비가동 정보
- 비가동 컬럼에 값이 존재하면 비가동 원인으로 해석한다.
- 비가동시간이 존재하면 해당 원인의 시간 값으로 사용한다.

#### (2) 부적합 정보
- 부적합 컬럼에 값이 존재하면 부적합 원인으로 해석한다.
- 부적합수량이 존재하면 해당 원인의 수량 값으로 사용한다.

#### (3) 불량 정보
- 불량 컬럼에 값이 존재하면 불량 원인으로 해석한다.
- 불량수량이 존재하면 해당 원인의 수량 값으로 사용한다.

### 3.4 데이터 해석 규칙

- 하나의 작업 단위는 1개의 Main Row와 0개 이상의 Detail Row로 구성된다.
- Main Row는 대표 생산 정보를 가진다.
- 원인 정보는 Main Row에도 존재할 수 있고, 추가 원인은 Detail Row로 확장될 수 있다.
- 동일 Key 그룹 내의 모든 비가동, 부적합, 불량 정보는 원인별로 누적 수집된다.
- Detail Row는 비가동, 부적합, 불량 중 하나 또는 여러 항목의 추가 원인 정보를 포함할 수 있다.

### 3.5 행의 연속성 특성

- 일반적으로 하나의 작업 단위(Main Row + Detail Row)는 엑셀 상에서 연속적인 행으로 구성된다.
- 즉, Main Row 바로 아래에 해당 작업의 Detail Row들이 순차적으로 배치되는 형태를 가진다.
- 그러나 이는 데이터의 일반적인 특성일 뿐, 필수 조건은 아니다.
- 파싱 시에는 행의 위치(연속성)가 아닌 Key 일치 여부를 기준으로 그룹을 구성해야 한다.


## 4. Group → Parsed Record 변환 규칙

동일 Key를 가지는 Main Row와 Detail Row들의 집합을 하나의 Parsed Record로 변환한다.

### 4.1 Parsed Record의 기본 원칙

- 하나의 Parsed Record는 하나의 작업 단위를 의미한다.
- Parsed Record는 1개의 Main Row와 0개 이상의 Detail Row를 통합한 결과이다.
- Parsed Record의 대표 정보는 Main Row에서 가져온다.
- Parsed Record의 원인 정보(비가동, 부적합, 불량)는 Main Row와 모든 Detail Row에서 누적 수집한다.

### 4.2 대표 필드 추출 규칙

다음 항목은 Main Row의 값을 사용한다.

- 일자
- Shift
- 공정 계층(Level1, Level2, Level3 등)
- 설비
- 제품
- 작업자
- 계획수량
- 실적수량
- 작업시간
- 효율
- C/T

### 4.3 원인 정보 누적 규칙

동일 Key 그룹 내의 모든 행에서 다음 항목을 수집한다.

#### 비가동 목록
- 비가동 값이 존재하는 경우 비가동 원인 항목으로 추가한다.
- 비가동시간 값이 존재하는 경우 해당 원인의 시간으로 저장한다.

#### 부적합 목록
- 부적합 값이 존재하는 경우 부적합 원인 항목으로 추가한다.
- 부적합수량 값이 존재하는 경우 해당 원인의 수량으로 저장한다.

#### 불량 목록
- 불량 값이 존재하는 경우 불량 원인 항목으로 추가한다.
- 불량수량 값이 존재하는 경우 해당 원인의 수량으로 저장한다.

### 4.4 Parsed Record 결과 구조

최종 Parsed Record는 다음 개념 구조를 가진다.

- key
- representative fields
- downtime_items[]
- nonconformity_items[]
- defect_items[]
- warnings[]

### 4.5 주의사항

- Main Row에 포함된 원인 정보도 반드시 누적 수집 대상에 포함한다.
- Detail Row는 대표 생산 정보를 가지지 않으며, 원인 정보 확장용으로만 사용한다.
- 동일 Key 그룹 내의 모든 원인 정보는 순서대로 보존하는 것을 원칙으로 한다.


## 5. 예외 및 보완 규칙

### 5.1 Main Row가 없는 경우

동일 Key 그룹 내에 Main Row가 존재하지 않는 경우:

- 해당 그룹은 비정상 데이터로 간주한다.
- Parsed Record는 생성하지 않는다.
- warnings에 기록한다.

### 5.2 하나의 행에 여러 원인 존재하는 경우

하나의 행에 비가동, 부적합, 불량이 동시에 존재할 수 있다.

- 각 항목은 독립적인 원인으로 분리하여 수집한다.
- 하나의 행에서 여러 리스트에 동시에 추가될 수 있다.

### 5.3 원인은 있으나 수량/시간이 없는 경우

- 원인 값이 존재하고 수량/시간이 없는 경우:
  - 수량/시간은 NULL로 저장한다.
  - 0으로 강제 변환하지 않는다.

### 5.4 인코딩 처리 규칙

- UTF-8로 먼저 시도한다.
- 실패 시 EUC-KR로 재시도한다.
- 둘 다 실패 시 오류 처리한다.

### 5.5 구분자 처리 규칙

- 기본 구분자는 콤마(,)로 처리한다.
- 필요 시 config에서 구분자를 지정할 수 있다.
- 자동 감지는 하지 않는다.

### 5.6 효율 컬럼 처리

- 효율 컬럼은 입력값 그대로 유지한다.
- 기본값이 1일 수 있으나, 강제값으로 처리하지 않는다.

### 5.7 Main Row가 2개 이상인 경우의 대표 필드 선택 규칙

동일 Key 그룹 내에 Main Row가 2개 이상 존재하는 경우:

- warnings에 기록한다.
- 대표 필드는 가장 먼저 등장한 Main Row의 값을 사용한다.
- 나머지 Main Row의 대표 필드는 무시한다.
- 단, 나머지 Main Row에 원인 정보(비가동, 부적합, 불량)가 존재하면 원인 정보는 수집 대상에 포함한다.

### 5.8 원인 중복 처리 규칙

동일 Key 그룹 내에서 동일한 원인 값이 여러 행에 반복될 수 있다.

예:
- 비가동: "작업변경"이 여러 행에 존재
- 불량: "찍힘"이 여러 행에 존재

처리 규칙은 다음과 같다.

- 원인 항목은 중복 제거하지 않는다.
- 각 행에서 수집된 원인 정보는 독립된 항목으로 그대로 유지한다.
- 동일 원인이라도 발생 횟수 및 수량/시간이 다를 수 있으므로 병합하지 않는다.
- 원인 리스트는 입력 데이터의 순서를 유지하여 저장한다.
